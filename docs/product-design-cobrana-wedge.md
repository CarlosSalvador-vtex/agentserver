# Design: Self-Service Agent Platform for WhatsApp Customer Service (Cobrança Wedge)

## Problem Statement

Companies doing customer service and debt collection via WhatsApp are locked into their
current AI agent providers because changing agent behavior (tone, scripts, escalation
rules) requires contacting a developer. This creates a scale ceiling: the company can't
iterate on their agent without vendor involvement, and the vendor monetizes this friction
as support tickets.

The target market (cobrança / debt collection companies in Brazil) has a defined
workflow: contact → proposta de renegociação → aceite/recusa. This workflow benefits
from an AI agent with a specific persona and domain skills. But the market doesn't need
"WhatsApp + AI" — Meta gives that away free. It needs **self-service iteration of agent
configuration without developer involvement.**

## Demand Evidence

- **Real person:** The founder's sister owns a debt collection company and currently uses
  a competitor platform. She is an identified first customer and potential reseller.
- **Stated pain:** "Para manipular o agente ela tem que conversar com o desenvolvedor —
  e isso não escala." Said about the current competitor, not a hypothetical.
- **Distribution signal:** Sister has expressed interest in reselling the product to
  other customer service companies — making her both user and channel.
- **Competitive gap observed directly:** Not a hypothesis. The founder has first-hand
  knowledge of how the competitor operates (developer-dependency is the lock-in).

## Status Quo

| Company type | Current approach | Cost |
|---|---|---|
| Debt collection (cobrança) | Pure human agents on WhatsApp | Scale ceiling: N calls/day = N human hours |
| SAC / customer service | Static workflow bot + human escalation | Frozen scripts, can't adapt to new cases |
| Both | Developer call required for any agent change | Hours-to-days delay per config iteration |

The competitor's business model is services disguised as software — every config change
is a developer ticket. Lock-in is the feature, not a bug.

## Target User & Narrowest Wedge

**Primary persona:** COO / Owner of a debt collection (cobrança) company in Brazil.
- Responsibility: recovery rate (taxa de recuperação de dívida)
- Consequence if unsolved: can't scale without hiring more agents; can't iterate without
  calling the developer; stuck with static behavior while market adapts
- First instance: the founder's sister (named, accessible, motivated)

**Narrowest wedge:** One company (sister's), one workflow (cobrança), one agent — but
she can change the agent herself on Thursday without contacting anyone. Measured by:
did she make a change and test it independently?

## Constraints

- Existing backend: Go/PostgreSQL/EKS — full multi-tenant infrastructure already works
- Existing frontend: React/TypeScript — Playground exists but disabled (skills were
  "created externally" — now revealed as the wrong call given user needs)
- WhatsApp integration: IM bridge already working
- Marketplace: skill and soul templates already exist system-wide
- No budget for new infrastructure; all changes must be additive to existing stack

## Premises

1. **(Revised)** Differentiator is self-service, no-dev-dependency, domain workflow
   agent — NOT generic WhatsApp chatbot. UI naming: "Personalidade do agente" +
   "Funcionalidades". Architecture internally: soul + skill.
   *Revision from cross-model review: "souls and skills" is architecture vocabulary,
   not product vocabulary. Sister thinks in conversation flows, not capability bundles.*

2. First sale is relational — via the founder's sister. No inbound marketing will
   produce the first customer.

3. Wedge = sister can change agent configuration Thursday without calling a developer.
   Success criterion: she makes a change, sends herself a test WhatsApp, and says
   "sim, é isso que eu queria."

## Approaches Considered

### Approach A: Playground Mínimo (48h prototype)
Re-enable Playground with stripped UI for non-devs. Name + Tone + Instructions → writes
SOUL.md. Deploy button + WhatsApp sandbox for immediate testing. Uses existing code.
Effort: S (1-2 days frontend only). Risk: Low. Would give real feedback in 72h.

### Approach B: Self-Service Onboarding Completo (chosen)
Full flow: company signs up → forks marketplace template → configures "Personalidade"
+ "Funcionalidades" via simplified UI → deploys → tests on WhatsApp → runs autonomously.
UI abstracts souls/skills with product naming. Effort: L (3-4 weeks, see breakdown below).
Risk: Med — building without early validation.

### Approach C: Managed Service First (agency → product)
Configure sister's agent manually using existing API. Observe every friction point in
production. Effort: S setup + M product. Risk: founder becomes the developer-dependency
he's trying to eliminate.

## Recommended Approach

**Approach B — Self-Service Onboarding Completo**, with a mandatory validation gate.

### Step 0 — Validation Gate (REQUIRED before any UI code)

Sit down with the sister. Show her the current system (or a Figma mockup of the planned
flow). Ask: "Mostra como você mudaria o que o agente fala quando o devedor diz que já
pagou." Watch. Don't help. Don't explain the architecture. Whatever she can't do becomes
the first screen to build. Gate output: a prioritized list of 3-5 screens ranked by
friction observed. No code until this list exists.

### Step 1 — Re-enable Playground as "Configurar Agente" (~3 days, frontend)

What "disabled" means technically: `App.tsx` has the Playground routes commented out
(marked `PLAYGROUND DISABLED`), and `TopBar.tsx` has the nav button commented. Re-enable
is: uncomment routes + nav, then rename labels in `Playground.tsx`,
`PlaygroundSoulEditor.tsx`, `PlaygroundSkillEditor.tsx` from "soul/skill" to
"Personalidade/Funcionalidades". No backend changes to soul/skill storage or endpoints.
Exception: Step 4 (deploy button) may require a backend cleanup path if sandbox creation
is not idempotent — investigate before declaring Step 1 complete.

### Step 2 — Marketplace fork to "Usar modelo de cobrança" (~3 days, frontend + template)

The fork flow exists (`forkMarketplaceSkill/Soul` in `api.ts`, `POST
/api/marketplace/{skills,souls}/{id}/fork`). What's missing: a cobrança template pair
in the system marketplace (workspace_id IS NULL). Template content spec:

**Cobrança soul template:**
- Name: "Agente de Cobrança"
- Persona: professional, empathetic, firm. Knows that the debtor has a reason for the
  delay. Goal: reach a payment agreement, not to pressure. Escalates to human if debtor
  refuses 3 proposals or requests to talk to a person.
- Tone: formal by default, adapts to debtor's register
- Language: Brazilian Portuguese

**Cobrança skill template:**
- Name: "Negociação de Dívida"
- Capabilities: present debt amount, propose payment plans (integral / parcelado),
  confirm agreement, send static payment link placeholder (text only — not a live
  integration; engineer should NOT add a payment gateway hook), handle "já paguei"
  response, handle "não posso pagar agora" response
- Escalation trigger: debtor says "quero falar com uma pessoa" — agent sends a fixed
  escalation message and stops responding. The escalation API contract (webhook, event,
  or human-takeover handoff) is a v2 decision. For v1: agent says "vou te conectar com
  um atendente" and the human follows up manually.

### Step 3 — Simplified editor UI (~5 days, frontend)

Map UI labels to architecture internals (never expose soul/skill vocabulary in UI):
- "Personalidade do agente" → soul editor (name, tone dropdown, instructions textarea)
- "Funcionalidades" → skill list with toggle per skill + config per enabled skill
- "Configurações" → composition-level settings (escalation phone number, working hours)

**Save semantics for forked entities:** When the user edits a forked soul or skill, the
edit updates the workspace-scoped fork in place (existing `PATCH /api/playground/souls/{id}`
and `PATCH /api/playground/skills/{id}` endpoints). No new version is created on save —
the fork IS the workspace's copy. This is consistent with the existing draft model.

### Step 4 — Deploy button from UI (~3 days, frontend + backend)

Currently: deploy requires `POST /api/workspaces/{wid}/sandboxes` with composition JSON
(API call only). The endpoint is authenticated with workspace member role. Verify that
the sister's account is granted member role at workspace creation time — this must be
part of the manual workspace setup checklist (see Decisions Required table).

Adding a "Publicar agente" button in the UI calls this existing endpoint. What to check:
does the composition hot-reload an existing sandbox or create a new one? If create: the
old sandbox must be deleted first (cleanup logic needed in the handler or frontend).
Effort depends on this answer — investigate before coding.

### Step 5 — WhatsApp test sandbox (~2 days, ops + backend)

The IM bridge routes messages by `im_channel_id` → `workspace_id`. A test sandbox needs
a WhatsApp number (BSP-provided) bound to the sister's workspace. Two options:
(a) dedicated number per workspace (scalable, costs per number), or
(b) shared sandbox number with per-workspace routing by initial keyword.
Option (b) is faster to implement for a single pilot customer. Caveat: Meta's WhatsApp
Business API has a 24-hour messaging window (business can only reply within 24h of last
debtor message). Keyword-based routing requires the IM bridge to parse the first message
before routing — this logic does not currently exist and must be scoped separately.
If keyword routing complexity is high, use option (a) (dedicated number) for the pilot
and defer multi-tenant routing to the second customer. Use option (a) when going to the
second customer regardless.

**Total effort estimate: ~19-24 days**
- Step 0 (validation gate): 1-2 days
- Steps 1-5: ~16 days (3+3+5+3+2)
- Buffer for Step 4 investigation (sandbox hot-reload vs create-new) + Step 5 routing
  complexity: 3-5 days

If Step 0 reveals a major pivot (e.g., sister thinks in flows not fields), add 5-7 days
to redesign Step 3 before building. This is not failure — it's the gate working.

## Success Criteria

1. Sister makes one agent configuration change independently (no developer contact) —
   measured within 2 weeks of Step 1 completion
2. Agent handles a debt negotiation conversation end-to-end on WhatsApp
3. Within 30 days of go-live: sister conducts one demo call to another company unassisted;
   outcome (interested / not interested) recorded by founder
4. Recovery rate measurable before vs. after deployment — baseline captured week 1 by
   asking the sister for her current monthly recovery rate; post-deployment measured via
   agent interaction logs (`agent_interactions` table) counting outcomes: agreement /
   refusal / escalation. No new instrumentation required for v1.
